Return existing notes for duplicate link creates

main · worktree /Users/anishthite/workspace/links · 2026-06-29 · status shipped · blast low

Summary

Goal
Make POST /api/notes idempotent for normalized sourceUrl creates.
Source
normalizeSourceUrl() already strips hashes, tracking params, case drift, and default ports.
Surface
POST /api/notes and POST /api/links.
Schema
No migration; keep idx_notes_source_url_normalized non-unique.
Open Qs
0.

API contract

Request

POST /api/notes
{ "sourceUrl": "https://example.com/post?utm_source=x#top", "text": "optional", "tags": ["optional"] }

Response — new link

201 Created
{ "note": NoteWire }

Response — duplicate link

200 OK
{ "note": NoteWire, "duplicate": true }

Algorithm

  1. Build the pending link row with buildLinkNoteInsert().
  2. Query notes.source_url_normalized = row.sourceUrlNormalized before insert.
  3. Return the newest existing joined note with duplicate: true; do not scrape again.
  4. Insert and schedule scrape only when no normalized match exists.

Decisions

IDChoice · Rationale
D-001Return existing unchanged. Avoid silent text/tag merges on a note the user did not open.
D-002Use app-level lookup over a unique index. Existing production duplicates could block a unique migration.
D-003Skip background extraction when ExecutionContext is absent. Direct app.fetch() tests only need the pending DB row.

Tradeoffs

IDPicked over Rejected · Reason
T-001200 + duplicate: true over 201. Callers can distinguish idempotent return without parsing note age.

Reversibility

R-001 · low. Remove the pre-insert lookup and the focused test.

Followups

  • L-001 · Add a unique partial index after cleaning existing duplicates if concurrent double-submits matter.

Definition of done

  • Duplicate normalized link returns the first note uuid instead of inserting a second row.
  • New link create still returns 201 and schedules extraction.
  • npm test -- tests/link-create-dedupe.test.ts passes.
  • npm run typecheck passes.
