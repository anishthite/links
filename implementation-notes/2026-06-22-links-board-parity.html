Decisions

IDChoice · Rationale
D-001Fork board wholesale first. Lowest-risk path to exact UI/backend/agent parity.
D-002Keep the core entity as notes internally. Reuses the board code with minimal churn; link behavior rides on extra columns.
D-003Store link metadata on notes as nullable source_* columns. Plain notes still work; links gain fetch/search/source actions.
D-004Use the existing board write panel and editor, then add a URL field and source buttons. Preserves muscle memory and visual parity.
D-005Expose /api/links as an alias, not a separate stack. External callers get a link-shaped path without duplicating handlers.

Tradeoffs

IDPicked over Rejected · Reason
T-001board fork + small adaptations over grow the MVP forward. Parity is faster by copying the proven app.
T-002nullable columns on notes over new links table + full port. Much smaller blast radius.
T-003saved extract text over live-fetch-only answers. Agent/search stay stable even if the source page changes.

Reversibility

R-001 · medium. Revert by redeploying the pre-parity Worker or deleting the new links-board-db; the old MVP database is untouched.
